> 版本：v1.0
> 日期：2026-04-11
> 作者：宋明璇

---

## 一、项目定位与愿景

### 一句话定义

**围绕单一商品期货品种，持续回答四个核心问题：**
1. 最近什么变了
2. 为什么变了
3. 这些变化是短期扰动还是中期趋势
4. 对期货和产业链股票分别意味着什么

### 项目最终产出，不是"数据展示"，而是"状态结论"

本系统最终产出 4 类结果：

**第一类：日频状态卡**
例如 PTA 当天输出：
- 基差走强还是走弱
- 仓单增加还是减少
- 现货强还是盘面强
- 当前更像"现实偏紧"还是"预期偏强"
- 结论偏多、偏空、偏中性，还是更适合做跨期/对冲

**第二类：事件时间轴**
把所有影响 PTA 的公司公告事件，按时间排成一条"产业事件流"：
- 扩产
- 投产
- 检修
- 停车
- 原料成本变动
- 下游订单变化
- 项目审批推进
- 业绩和利润变化

**第三类：趋势归因报告**
不是逐条列公告，而是回答：
- 过去 1 个月，市场是变得更好还是更坏
- 改善来自供给端、需求端、成本端，还是库存端
- 改善是现货层面的，还是预期层面的

**第四类：交易翻译结果**
同一条结论，要翻译成两种语言：
- 对 PTA 期货意味着什么
- 对产业链股票意味着什么

**也就是说，项目本质上是一个"行业状态机 + 交易解释器"。**

## 二、总体架构：三层引擎，最后合成一个结论

### 核心三层引擎

```
┌─────────────────────────────────────────────────────────────┐
│                    三层引擎                                  │
├─────────────────────┬─────────────────────┬────────────────┤
│   数值因子引擎       │   文本事件引擎       │  时间序列综合引擎 │
│   （数据层）         │   （事件层）         │  （趋势层）      │
│                     │                     │                 │
│  现货、基差、期货    │  公告、年报、调研    │  供给/需求/成本  │
│  仓单、数值因子      │  抽成标准化事件      │  边际趋势判断    │
└─────────────────────┴─────────────────────┴────────────────┘
                              ↓
              ┌───────────────────────────────┐
              │      交易翻译引擎（Agent D）    │
              │  期货含义 + 产业链股票含义      │
              └───────────────────────────────┘
                              ↓
                    ┌─────────────────┐
                    │   日频状态卡      │
                    │  事件时间轴       │
                    │  周报/月报        │
                    └─────────────────┘
```

### 1. 数值因子引擎

负责处理市场数据，把数据变成结构化事实。

**输入：**
- 现货价格
- 基差（主力/近月）
- 期货主力/近月价格
- 仓单
- 后续再补：社会库存、加工费、月差、持仓

**这一层的任务不是预测，而是解释**，输出结构化事实：
- 基差 5 日/20 日变化
- 基差分位
- 仓单变化方向
- 仓单是否进入异常区间
- 期现关系是升水还是贴水
- 期限结构是 back 还是 contango

**数据来源：**
- Tushare：`fut_daily`（期货日线）、`fut_mapping`（主力/连续合约映射）、`fut_wsr`（仓单日报）
- 100ppi.com：现货价格

### 2. 文本事件引擎

负责处理公司公告和报告，抽成标准化事件。

**输入：**
- 定期报告（年报、半年报）
- 临时公告
- 调研记录
- 业绩说明会问答
- 环评/项目批复
- 海关、行业统计等外部信息

**这一层不应直接输出长段摘要，而要先抽成标准化事件。**

标准化事件格式：
- 事件类型：扩产 / 检修 / 停车 / 投产 / 订单变化 / 成本变化
- 公司
- 所属产业链环节
- 涉及产品
- 产能规模
- 生效时间
- 对 PTA 的影响方向
- 证据片段
- 置信度

**数据来源：**
- 东方财富（已接入）
- 巨潮资讯
- 证监会法定披露文件
- 生态环境部环评公示
- 海关统计数据

### 3. 时间序列综合引擎

**这是最关键的一层。** 它不分析单个数据点，也不分析单篇公告，而是把两边结果按时间拼起来，回答：
- 过去两周供给边际是收缩还是扩张
- 过去一个月需求边际是改善还是走弱
- 近期成本推动增强还是减弱
- 数据和公告之间是否互相验证
- 市场现实和未来预期是否一致

## 三、4 个专业化 Agent

> 不要一个大 Prompt 全包，拆成 4 个专业化 Agent

### Agent A：数值解释 Agent

**输入：** 结构化 JSON（不是原始表）

输入示例：
```json
{
  "main_basis": 50,
  "main_basis_5d_change": 10,
  "main_basis_20d_change": -5,
  "near_basis": 45,
  "warehouse_receipt": 50000,
  "warehouse_receipt_change": 3000,
  "spot_price": 5800,
  "fut_close": 5750,
  "time_spread": 20,
  "back_or_contango": "back"
}
```

**输出格式固定为：**
```json
{
  "today_summary": "基差走强，仓单增加",
  "core_interpretation": "现货强于期货，期限结构收紧",
  "direction": "偏多/偏空/中性",
  "confidence": "高/中/低",
  "counter_evidence": ["风险点1", "风险点2"]
}
```

**职责：不是预测价格，而是解释当前状态。**

---

### Agent B：单篇公告抽取 Agent

**输入：** 一篇公告全文（title + full_text + notice_date + stock_name + stock_code + f_node_name）

**输出：结构化事件，不是摘要。**

Prompt 重点是**抽字段**，强制输出：
```json
{
  "event_type": "扩产/检修/停车/投产/订单变化/成本变化",
  "company": "公司名",
  "chain_position": "上游/中游/下游",
  "product": "涉及产品",
  "capacity_mtpa": 产能数字或null,
  "supply_impact": "增加/减少/不变",
  "demand_impact": "增加/减少/不变",
  "effective_date": "生效时间",
  "time_horizon": "短期/中期/长期",
  "evidence": "关键证据片段",
  "confidence": "高/中/低"
}
```

**关键原则：事实和推断强制分离。**

---

### Agent C：事件时间轴 Agent

**输入：** 某品种过去一段时间的 `event_fact` 记录

**目标：不是复述事件，而是判断趋势。**

回答：
- 供给扩张在加速还是放缓
- 检修扰动是临时还是持续
- 下游需求事件是改善还是恶化
- 公告整体是在验证市场强势，还是在削弱它

**输出：**
```json
{
  "supply_trend": "扩张加速/扩张放缓/收缩/平稳",
  "demand_trend": "改善/走弱/平稳",
  "cost_trend": "增强/减弱/平稳",
  "data_event_consistency": "一致/不一致",
  "reality_vs_expectation": "现实强于预期/预期强于现实/一致",
  "key_drivers": ["主要驱动因素1", "主要驱动因素2"]
}
```

---

### Agent D：交易翻译 Agent

**输入：**
- 数值因子结论（Agent A 输出）
- 事件趋势结论（Agent C 输出）

**输出：**

```json
{
  "fut_interpretation": {
    "direction": "看涨/看跌/震荡",
    "reasoning": ["理由1", "理由2"],
    "risk_points": ["风险点"]
  },
  "stock_interpretation": {
    "direction": "看多/看空/中性",
    "affected_companies": ["公司1", "公司2"],
    "reasoning": ["理由"]
  },
  "consistency": "期货和股票方向一致/不一致",
  "reality_weight": "哪个更偏现实，哪个更偏预期"
}
```

**这是非常必要的，因为同一个产业事件，对期货和股票不一定同方向。**

## 四、8 张核心数据表

> 数据库围绕"实体—事件—状态"设计，不只是价格表和公告表

### 1. commodity_master
记录品种信息（ PTA、PX、MEG、短纤、瓶片等）

### 2. company_master
记录产业链公司，包含：
- 公司代码、公司名称
- 是否上市、所属交易所
- 主营方向、所处环节

数据来源：Tushare `stock_basic`

### 3. company_chain_exposure
**很关键的一张**，不是只写"公司属于 PTA 产业链"，而要写：
- 上游 PX 暴露程度
- 中游 PTA 暴露程度
- 下游聚酯暴露程度
- 一体化程度
- 利润弹性
- 产能暴露程度

### 4. price_factor_daily
存数值因子，存：
- spot（现货价格）
- settle / close（结算价/收盘价）
- main_contract / near_contract（主力/近月合约）
- basis_main / basis_near（基差）
- basis_rate（基差率）
- basis_percentile（基差分位）
- warehouse_receipt（仓单量）
- warehouse_receipt_change（仓单变化）
- contango/back（期限结构）

### 5. disclosure_doc
存公告或报告元数据：
- company、doc_type、ann_date、title
- source、url、raw_text
- source_level（证据等级）

### 6. event_fact
**文本分析后的核心结果表。** 一条公告可能抽出多条事件。
- event_id、company、commodity、chain_position
- event_type、product、capacity
- direction、effective_date
- evidence_text、confidence（1-5级证据）
- source_doc_id

### 7. commodity_event_timeline
把所有事件映射到品种层。某家聚酯龙头扩产，虽然公告主体不是 PTA 公司，但仍影响 PTA 需求，所以要被映射进 PTA 的事件时间轴。

### 8. commodity_state_daily
**最终输出表**，每天每个品种一条记录：
- 供给边际评分
- 需求边际评分
- 成本边际评分
- 库存/仓单边际评分
- 文本情绪评分
- 综合结论
- 风险点
- 推荐观察方向

## 五、证据等级制度（核心原则）

### 5 级证据

| 等级 | 类型 | 示例 |
|------|------|------|
| 一级 | 交易所数据、法定公告、年报、半年报、正式项目批复 | Tushare期货数据、巨潮公告、证监会规范、郑商所PTA业务细则 |
| 二级 | 机构调研、业绩说明会、公司交流纪要 | 投资者关系活动记录表 |
| 三级 | 环评、项目审批、公示、海关和统计数据 | 生态环境部建设项目环评、海关统计数据在线查询平台 |
| 四级 | 媒体报道、行业快讯 | 我的钢铁网、隆众资讯 |
| 五级 | LLM 推断 | Agent 生成的推断 |

### Agent 硬规则

**事实和推断强制分离。**

例如：
- 事实：某公司公告 PTA 装置检修 15 天
- 推断：若无其他装置补量，短期 PTA 供给边际偏紧

**不要把推断写成事实。**

## 六、公告/年报 Agent 的正确用法

证监会 2025 年修订的《公开发行证券的公司信息披露内容与格式准则第2号——年度报告的内容与格式》自 2025 年 7 月 1 日起施行。规则要求公司在"管理层讨论与分析"中披露：
- 业务情况、主要产品及用途、经营模式、市场地位、竞争优劣势、主要业绩驱动因素、行业阶段和周期性、政策影响
- 重大投资项目、资产购买、兼并重组、在建工程、研发项目、关键业绩指标及其变化原因
- 年报应在会计年度结束后 4 个月内披露

**公告/年报 Agent 不应泛泛总结，而应围绕这些固定问题抽取：**
1. 业务有没有变化
2. 产品结构有没有变化
3. 行业景气的描述有没有变化
4. 供应商、客户、库存、销量有没有变化
5. 新产能、在建工程、重大投资有没有变化
6. 财务变动背后的业务根因是什么

**这会让文本分析非常"规整"，而不是写作文。**

## 七、PTA 样板落地：两条主线

### A. 数据主线
先跟踪：
- 现货价格（100ppi）
- 主力基差 / 近月基差
- 主力合约价格
- 仓单量、仓单变化
- 月差（back/contango）
- 后续补：社会库存、PX-PTA 加工差、聚酯利润

**第一版 Agent 每天只回答：**
1. 市场现在是现货强还是盘面强
2. 仓单变化在验证还是否定盘面
3. 当下更像供需偏紧还是宽松
4. 是现实驱动还是预期驱动

### B. 文本主线
围绕 PTA 产业链建立公司池：
- 上游原料公司（荣盛石化、恒逸石化、恒力石化、东方盛虹）
- PTA 生产公司
- 下游聚酯、瓶片、长丝、短纤公司（桐昆股份、新凤鸣、万凯新材、三房巷）
- 相关项目公司

**抓三类文本：**
1. 定期报告（年报、半年报）
2. 临时公告（投资公告、检修公告、业绩预告）
3. 调研/交流记录

**单篇公告抽事件，随后按月度或季度聚合，形成事件趋势判断。**

例如：
- 多家公司同时公告检修 → 供给偏紧
- 多家公司持续披露扩产/环评推进 → 中期供给压力上升
- 下游公司反复提到订单转弱、库存上升 → 需求偏弱
- 多家公司提到原料上涨无法顺利传导 → 中下游利润承压

    └── 股票数据（已有）

## 十二、关键设计决策

### 12.1 为什么要分 4 个 Agent？

| Agent | 输入 | 输出 |
|-------|------|------|
| A 数值解释 | 数值因子 JSON | 当前状态解释 |
| B 公告抽取 | 单篇公告全文 | 结构化事件 |
| C 时间轴 | 多条 event_fact | 趋势判断 |
| D 交易翻译 | A结论+C结论 | 期货+股票双路径 |

**分层的目的是让每次 LLM 调用专注一个任务，减少幻觉，提高输出稳定性。**

### 12.2 为什么要先抽事件，再做时序分析？

**结构化提取 > 无结构输入。**
- 输出可存储、可查询、可比较
- 时序分析有了一致的数据结构
- 避免 LLM 每次都重复读原始长文本
- 事实和推断分离，证据链清晰

### 12.3 为什么要区分期货和股票两条路径？

**同一个产业事件，对期货和股票不一定同方向。**

例如：
- PTA 扩产 → 对期货偏空，但对聚酯龙头股票可能偏多（规模效应）
- 下游需求疲软 → 对期货偏空，对现金充裕的上游公司可能中性

## 十三、项目本质

> 把离散的市场数据和离散的公司事件，统一放进同一条时间轴里，再把它们转成对品种供给、需求、成本、库存、预期的连续判断。

这件事一旦做成，你得到的就不是一个资讯聚合器，而是一套真正可用的**品种研究底座**。
